Accessibility examples

Skip links

It is useful to have a link to skip to the main content of a page. This makes your page more keyboard and screen reader user friendly.

Put the following as the first thing in body (where "main" should be whatever the id of your main element is):

<a href="#main">Skip to main content</a>

You can then style your link to be hidden with absolute positioning and only show it when it's in focus (:focus pseudoclass).

Designing with descriptive links and headings

Use descriptive text for links and headings.

Consider:

Read more
Read more about penguins

The second link is more descriptive and is more user-friendly for screen reader users. If you have no choice but to use the more generic "Read more" consider adding aria-label or a title attribute. For example:

<a href="#" title="Find out more about penguins">Read more</a>

The result:

Read more

Using the title attribute is useful in this case for both sighted mouse users and screen reader users.

Alt text

Alt text for images should be descriptive and provide context. An img without an alt attribute is invalid HTML.

Try with a screen reader:

In 99% of cases, images should have alt text. This means that the alt attribute value should NOT be blank. The exception where you can omit alt text is if the image is not significant and might just be purely decorative. In that case, you would still have the alt attribute (but with an empty value) and also use aria-hidden="true" to tell screen readers to skip the image. For example:

          <div>
            <img src="./img/checkmark.png" alt="" aria-hidden="true" width="50">
            Correct
          </div>

This would result in:

Correct

NOTE: Free in-browser screen readers might still read elements even with aria-hidden="true".

Alternatively, if something is purely decorative, it might make more sense to insert using CSS.

Image-only or icon-only links

For image- or icon-only links, there should be hidden text for screen reader users at the very least. You can add hidden text using the aria-label attribute.

<a href="#"><i class="fa-solid fa-envelope" aria-label="Email"></i></a>

Alternatively, use the title attribute to also provide context to mouse users.

Designing with colour

Make sure your text has good contrast against the background. Try using the WAVE accessibility browser extension to help identify colour contrast issues. Low contrast means that users with vision impairment will have more trouble reading your text (even if you think it's legible).

Also avoid using colour alone to convey meaning.

For example, avoid doing the following:

Veritas tools:

  • work out of the box
  • needs little fettling
  • a little pricy

You can use colours but make sure that the meaning is clear even without the colours. For example:

Tool brand Pros Cons
Veritas
  • works out of the box
  • needs little fettling
  • can be pricy

Menus

When you have a clickable element which triggers something, use a button instead of a clickable div. A button's purpose is to trigger some action upon clicking and is natively interactive. A div has to be made interactive but is not fully accessible and interactive out of the box. Try interacting with both (below) using a keyboard. Tab to each and hit [Enter] to "click".

You cannot reach the div because div is not natively an interactive element (even if you add onclick). You can add an attribute to make it reachable by tabbing (add tabindex="0") and therefore making the div register as interactive, but this is an extra step that is unnecessary if you just use button in the first place.

It is also useful to use aria-label to indicate what type of menu the nav is.

Impact of text choices

Text choices can affect screen reader accessibility. Try: CONTACT US vs contact us

Forms with focus styling

In addition to general focus styling, you can also use the :focus-within pseudoclass to highlight forms with a field selected.

Link "buttons"

Sometimes you want links to look like buttons. That is just a matter of CSS styling but you should still use the "a" element alone. In the case of an element which looks like a button but behaves like a link, just use a link and style with CSS to look like a button.

Do not use a button or nest a button inside a link or vice versa. Nesting a link in a button or vice versa is not user-friendly for keyboard users or screen reader users.

Tabbing once selects the button. A user would expect to be able to hit [Enter] and go to the link destination but nothing happens. This is because, on first tab, the user focuses on the button. It takes a second [Tab] to focus on the link. This is confusing (and it's invalid HTML too).
